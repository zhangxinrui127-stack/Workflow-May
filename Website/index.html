Cognition · 认知层

理解和认知层

用 AI 之前先想清楚：什么任务该外包、什么任务必须留在自己手上。

1 · 应该用 AI 来做什么？目前能力边界如何？+
  • 工作流的流程化、标准化、proofread 与审阅流程：把研究、撰写、校对number-verification skill、交叉审阅这些可重复的环节固化成一条可执行的流水线交给 AI 串起来跑；人来负责定义这条流水线，并确保它持续符合专业工作的标准与要求。
  • 个人知识沉淀：embedding RAG / 或者直接存 md 文件，做自己的 Agent 友好型数据库。
  • 能 vs 不能：以前需要找 vendor 写的爬虫、需要专家访谈才能 cover 的技术性部分（架构理解、技术栈对比、benchmark 解读），现在都可以通过 AI 推进 — 边界在被持续推开。
  • 快 vs 慢：以前需要"搬运 + 重复性"的搜集资料工作（多源信息抓取、表格清洗、初稿撰写、长文 summarize）都可以自动化完成，把时间留给真正不可替代的判断与决策。
2 · 从 Agent 的结构，来理解 Agent 的能力边界？+
  • 执行环境：Claude Code 能直接读 / 写本地文件，有完整的文件系统权限 + 命令执行权限。
  • 权限 / 工具链：可以接 GitHub、MCP、Skills 等扩展工具链 — GitHub →
  • 上下文管理：可以自己写 projects 里的 md 文件记录进度，也可以 /compact 主动压缩。
3 · AI 目前明显不能替代的部分，如何人机协同？+
  • 提供信息源、寻找资源，包括寻找工具本身。
  • 判断（尤其是方向性的、需要体感的、不一定是在语言边界里的判断，例如投资）。所以人无法从这个过程里 off 出去。
Tips · 其他经验

其他 Tips

长期使用 AI 写研究稿和做 Mapping 沉淀下来的工程化小经验。

1 · 输入多比输出多更重要+
要提高 AI 产出质量，需要的是输入更多信息（包括问能够提升置信空间的问题），包括更相关的参考文献和思路更清晰的提纲，还有人看到产出物后给出的具体修改意见。
2 · 每一步可拆解、有中间结果+
每一步可拆解、有中间结果，才能控制 context 上下文和方向。
3 · 先让 AI 生成 requirement prompt，再用它去执行+
想清楚功能 / 目标，让 AI 生成 requirement prompt；然后用此 prompt 去发起任务。对于渲染 rendering 类工作，最好是有模板来复刻，否则需要写很长的 PRD 才能 harness 住格式。
4 · 没有什么工程问题是 Codex 开十几个子代理解决不了的+
没有什么工程问题是 Codex 开十几个子代理解决不了，而非得在一个群聊里 @ 好几个 Agent 来完成的（吐槽 Slack）。
5 · 一些喜欢调用、用来延展 Agent 能力的工具+

对于通用类需求，我会优先找合适的产品 / 插件 / skills 接进来，让 Agent 的能力被自然延展。

  • 信息来源：https://aihot.virxact.com/
  • PPT：把 Grid、边距、占位符、字体、色值规定在 skills 里，让 python-pptx 等引擎去执行。
  • Word：python-docx 去执行，核心是格式 — 已经通过 perplexity 验证过它可以做模板复刻。
  • 对个人来说，"收集 / 处理信息" 和 "做判断 / 思考" 才是真正核心的工作，这部分的延展工具我会持续收集与替换。
