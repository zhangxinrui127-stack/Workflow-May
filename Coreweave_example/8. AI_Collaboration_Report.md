## 一、工作流总览

> 详见附件 `CoreWeave_AI_Workflow.png`（高清流程图）

### 整体架构：6 阶段、3 角色协同

```
PHASE 0  人工准备 → 任务拆解 → 研究命题手写
PHASE 1  多源情报采集（5 个 AI 工具 + 3 类数据源 → NotebookLM 汇聚）
PHASE 2  Claude 生成研究框架 → 人工调整 → 并行研究 → 8 模块初稿
PHASE 3  Claude 对抗审校 → 人工深度追问 → 估值修正 → 终稿
PHASE 4  协作设计 pptx-research Skill → 3 轮人工反馈 → 44 页 Deck
PHASE 5  Claude 3-Statements-Ultra → Bug 修复 → 人工假设验证 → 模型终版
PHASE 6  Deck ↔ Model 交叉验证 → Thesis Tone 对齐 → 人工最终签收
PHASE 7  Claude 读取全部 Session 记录 → 生成工作流图 + AI 协作报告 → 5 轮人工迭代
```

### 核心设计原则

| 原则 | 实施方式 |
|------|----------|
| **人工主导方向，AI 执行规模化** | 研究框架由人工手写核心命题，AI 扩展为 100+ 细分问题并执行研究 |
| **双源交叉验证** | NotebookLM（一手资料）与 Web 搜索（实时数据）并行查询，交叉校验 |
| **对抗式纠错** | 生成 Session 与 Review Session 分离，4 个并行 Agent 独立审核 |
| **人工决策节点** | 每个 Phase 末端设置人工审阅/追问环节，保留关键判断权 |
| **Token 效率优化** | 将大量一手资料导入 NotebookLM 而非直接投喂 Claude，节约 token 成本 |
| **渐进式交付** | Markdown 中间产物 → 人工确认 → 再渲染为 PPTX/XLSX，最小化返工 |

### 2.3 NotebookLM 的战略角色

**为什么用 NotebookLM 而非直接给 Claude？**

| 维度 | 直接投喂 Claude | 通过 NotebookLM 中转 |
|------|----------------|---------------------|
| Token 成本 | 每次对话消耗全量 token | NLM 免费存储，Claude 仅消耗查询+回答 token |
| 交叉索引 | 受限于单次 context window | NLM 可跨 30 份文档交叉引用 |
| 数据可靠性 | 一手资料与搜索结果混合 | 一手资料独立存放，数据层级清晰 |
| 复用性 | 每个新 Session 需重新上传 | 持久化存储，跨 Session 可用 |

**执行策略**：

| 策略 | 具体做法 |
|------|----------|
| 双源并行 | 每个 Module 同时向 NLM 和 Web 发起 3-4 个查询 |
| 跨模块并行 | Module 4+5 同时研究；Module 6+7 同时研究 |
| 信息源分工 | NLM → 财务数据/合同条款/管理层背景；Web → 市场规模/竞对/GPU 价格 |

**工具调用统计**：

| 工具 | 调用次数 | 用途 |
|------|----------|------|
| `notebooklm ask` | 12 | 查询 10-K/白皮书/研报中的一手数据 |
| `WebSearch` | 10 | 补充实时市场数据、竞对、分析师观点 |
| `Write` | 8 | 保存 8 个模块 Markdown 报告 |
| `TaskCreate/Update` | 24 | 任务管理（8 模块进度跟踪）|

**NotebookLM 查询设计模式**：
```
notebooklm ask "[Context Frame]: Provide detailed [topic].
Specifically address:
1) [Specific question with quantitative anchor]
2) [Specific question with comparison frame]
3) [Specific question with time series request]
..."
```

设计原则：
- 每个查询含 5-10 个编号子问题 → 强制结构化输出
- 包含定量锚点（"$88B RPO", "67% FY2025"）→ 引导引用具体数据
- 使用对比框架（"vs hyperscaler"）→ 生成可比分析
- 请求时间序列（"2023-2025 by quarter"）→ 获取趋势数据

**阶段产出**：8 份 Markdown 研究报告，~23,500 字，79 张表格

### PHASE 3：对抗审校与人工深度追问

#### 人工阅读 & 审校方向制定

在交给 Claude 做自动化审校之前，**人工先完整阅读了 8 个模块的 Markdown 初稿**，识别出需要重点审查的方向和关键问题，包括：
- 数字口径是否一致（cash vs accrual capex 等）
- 关键假设是否有实证支撑（债务融资比例、收入预测）
- 多空观点是否平衡（是否过于偏向 Bull Case）

基于人工判读结果，向 Claude Review Session 给出明确的审校指令。

#### 审校设计：独立 Session + 并行 Agent

**核心理念**：生成报告的 Session 与审校 Session 完全分离，避免 confirmation bias。人工先读、人工给方向，AI 执行规模化校验。

**Round 1 — 数字校对 + 逻辑审查**

**Prompt**：
> 对 8 个研究模块执行系统性审校：（1）数字校对——逐一核实全部定量数据，标注来源（SEC Filing / 研报 / 推算）和可靠性评级；（2）逻辑审查——检查各模块论证链条的完整性、假设的合理性和跨模块数据一致性，对每个审查点给出修正建议。产出：数字校对报告 + 8 份 Reviewed Module。

**执行**：启动 4 个并行研究 Agent——
1. 财务数据 Agent：核实收入、EBITDA、capex、EV、估值倍数
2. 合同/客户 Agent：核实客户合同金额、backlog、RPO
3. 行业数据 Agent：核实市场规模、竞对数据、GPU 价格
4. 治理 Agent：核实管理层背景、持股结构、董事会

**发现**：

| 严重级别 | 数量 | 代表性问题 |
|----------|------|------------|
| 严重 | 3 | EV 双重扣减现金（$70.3B→$79.9B）；FY2023 收入混淆 |
| 中度 | 7 | FCF 口径混淆；Hyperscaler capex 矛盾；利息支出偏高 |
| 轻微 | 3 | 来源标注不精确 |

**Round 2 — 49 个逻辑审查点二次验证**

**Prompt**：
> 逐一回顾 reviewed 文件夹中各 Module 的逻辑审查标注，将每个审查点作为独立专题进行二次事实收集与结论判定，产出 key-questions-validation 文档。重点深挖 Module 5 中的三个关键假设：
> 1）FY2027+ 债务融资比例的推导依据——有什么历史数据和 covenant 约束可以支撑？
> 2）利息支出对美联储降息路径的敏感性——SOFR 变动 100bp 对年化利息的影响量级？
> 3）FY2026 之后的收入增长依赖新签合同，如何基于各大客户的算力支出规划做底层预测？

**执行**：再次启动 4 个并行 Agent（逻辑提取、Fed 利率、客户算力、债务结构）

**Round 3 — 人工追问驱动的关键修正**

人工提出两个 Claude 无法自发产生的追问：

> ① FY2027+ 融资比例向股权倾斜——实际可行吗？  
> ② Capex 放缓对估值的影响：EBIT 转正需要 capex 放缓，但这是否会被解读为增长见顶？

**修正结果**：

| 指标 | 修正前 | 修正后 |
|------|--------|--------|
| Base Case 目标价 | $179 | $155 |
| 概率加权期望值 | $168 (+42%) | $139 (+17%) |
| 12 月折现期望值 | — | ~$121 (≈华尔街共识) |
| Distress 概率 | 5% | 10% |

**阶段产出**：
- `CRWV_数字校对报告.md`
- 8 份 Reviewed Module
- `key-questions-validation.md`（49 个审查点 + 3 专题 + 2 追问）
- `CRWV_Investment_Research_Memo_Final.md` (v2)

#### 4A — pptx-research Skill 协作设计

由于市面上没有适合「高密度投研/咨询型」PPT 的 AI 工具，与 Claude 从零协作设计了 `pptx-research` Skill。

**Prompt（核心需求描述）**：
> 协作设计一个面向投研/咨询场景的 PPTX 生成 Skill（命名为 pptx-research）。核心定位：内容强管控、高信息密度、多图表、非路演型 deck。Skill 加载后需先引导用户输入完整报告内容（doc/md 文件），依次确认语言、大纲、Storyline、Key Visuals、格式后再渲染，每个章节渲染后需用户确认才继续，最小化返工。

**Skill 架构**：
- 7 步状态机（Step 0-5 + Step 3b 逻辑审查）
- 9 个全局规则（G1-G9）
- 断点恢复机制（`_deck_state.json`）
- 12 种布局模板 + 9 种原生图表配方

**3 轮人工反馈与校准**：

| 轮次 | 问题 | 修正 |
|------|------|------|
| Round 1 | P5-P8 内容太少；表格边框多余 | 写入 G8（最低内容密度）、G9（对称布局）|
| Round 2 | 内容严重溢出 | 上传 BCG template.pptx 做参考，校准 commentary 密度 |
| Round 3 | 风格太偏 Roadshow；字体不统一 | 去掉巨大数字展示，改为分析评估表格；字体最小 12pt |

### PHASE 5：三表模型构建

#### 开源工具发现与整合

`3-statements-ultra` **并非 Claude 内置功能**，而是通过社交媒体平台和 GitHub 筛选发现的一个开源 Skill，由某 PE（Private Equity）从业者开发并公开发布。该 Skill 针对机构级三表模型（IS/BS/CF）的构建场景，支持完整公式联动、多频率自适应（季度/年度）、多会计准则（IFRS/US GAAP/中国准则），并内置 QC Suite 自动化检查。

**发现路径**：社交媒体推荐 → GitHub 仓库评估（Star 数、代码质量、文档完整度）→ 本地安装配置 → 接入 Claude Code Skills 系统

这一步体现了**善用公开工具链**的能力——在 AI 生态中，不仅使用平台原生功能，还主动发现和整合社区开源资源，将专业领域的最佳实践（PE 从业者的建模规范）引入自己的工作流。

#### 使用开源 `3-statements-ultra` Skill 构建模型

**Prompt**：

**执行过程**：跨 5 个 Session（A→E），包括：
- Sessions A-D：Raw Data → Assumptions → IS/BS/CF 完整联动
- Session E：Returns + Cross_Check + Summary + QC Suite
- Bug Fix：BS_COL_MAP off-by-one 关键修复
- 4-Phase Enhancement：Revenue Build-Up / Check Formulas / Scenarios / Formatting

**人工识别的 4 个问题**：
> 1）IS/BS/CF 三表的格式与 Skill 规范不一致（字体、配色、冻结窗格等）

## 四、人工介入关键节点总结

| 节点 | 人工做了什么 | 为什么 AI 做不到 |
|------|-------------|------------------|
| **选题 & 研究命题** | 基于自身对行业的理解手写核心研究问题 | AI 缺乏对"什么问题最重要"的判断力 |
| **信息源筛选 & 导入** | 挑选高质量资料导入 NLM | AI 无法判断资料的可信度层级 |
| **研究框架审阅** | 确认模块覆盖面和优先级 | AI 倾向于面面俱到，缺乏聚焦 |
| **报告初稿阅读 & 审校方向** | 先读报告，再给 Claude 明确审校方向 | AI 自行审校容易遗漏关键问题 |
| **Phase 3 追问** | 提出债务融资悖论、Capex 放缓悖论 | 这是需要投资经验的非显而易见问题 |
| **估值判断** | 确认修正后的目标价合理 | AI 可以计算但无法对"合理性"做最终判断 |
| **模型假设验证** | 核验 GPU fleet、ARPU、客户级收入方向 | 关键假设需要人工判断 business sense |
| **PPT Skill 风格校准 (4A)** | 3 轮反馈：密度、溢出、风格 | AI 无法判断"什么是机构级专业度" |
| **Storyline 逐页确认 (4B)** | 逐页审阅 title + outliner + key message | 结论导向的标题需要投资判断 |
| **逐章渲染确认 (4B)** | 每章渲染后 QA 确认再继续 | 信息呈现方式是主观的传达效率判断 |
| **McKinsey 截图对比 (4C)** | 上传 13 张真实 McKinsey 报告截图做对比 | AI 无法自主发现与机构标准的视觉 gap |
| **数据根因追溯 (4D)** | 要求先查清为什么写错再修改 | AI 默认直接替换，不追溯根因（根因追溯发现了版本混淆和认知盲区）|
| **口径标注 (4D)** | 指出 backlog/debt/cash 需标注计算口径 | AI 无法判断哪些口径差异需要向读者说明 |
| **PPT 格式 & Wording 手动调整** | 逐页调整格式、措辞、对齐、footnote | AI 渲染精度有限，机构级文档需人工最后一公里 |
| **Thesis Tone 对齐** | 确认最终投资结论应审慎而非乐观 | 投资结论的 tone 是主观判断 |

## 七、反思与局限

### 7.1 AI 做得好的地方
- **规模化信息处理**：12 次 NLM 查询 + 10 次 Web 搜索，覆盖面远超人工可能
- **结构化输出**：每个模块自动生成表格、KPI Dashboard、比较矩阵
- **并行执行**：4 个 Agent 同时核查不同维度，效率极高
- **一致性保障**：自动化 Cross-Check 和 QC Suite 确保数字一致

### 7.2 AI 做得不够好 / 需要人工干预的地方
- **初始估值偏乐观**：AI 生成的报告倾向于 Bull Case，需要人工追问才能修正
- **风格判断**：PPT 风格需要 3 轮人工反馈才校准到机构级水平
- **"不会问的问题"**：Capex 放缓悖论、融资结构可行性等非显而易见问题需要人工提出
- **投资论点 Tone**：AI 自动生成的 Model Summary 偏乐观，需人工指导对齐

### 7.3 善用公开工具链的方法论

本项目的工具链并非全部来自单一平台，而是**主动整合了多个生态的最佳资源**：

| 来源 | 工具 | 发现方式 |
|------|------|----------|
| Claude 原生 | Claude Code (Opus 4.6)、Web Search | 平台内置 |
| Google 生态 | NotebookLM、Gemini Deep Research | 跨平台组合使用 |
| 中国 AI 平台 | 元宝（腾讯）、豆包（字节）、Grok（X） | 针对中文信息源的差异化选择 |
| GitHub 开源社区 | `3-statements-ultra` Skill | 社交媒体推荐 → GitHub 筛选 → 本地安装 |
| 用户自建 | `pptx-research` Skill | 与 Claude 协作从零设计 |

**关键认知**：AI 工具的价值不仅在于单个工具的能力，更在于**组合使用和工具发现**的能力。在 AI 开源社区中，专业从业者贡献的 Skill/Plugin 往往比通用工具更贴合垂直领域需求（如 PE 从业者设计的三表模型 Skill 内置了行业标准的 QC 规范），善于发现和整合这些资源是 AI 时代的核心竞争力之一。

### 7.4 方法论可复用性
本次建立的工作流可直接复用于其他公司的投资研究，核心可复用模块包括：
- NotebookLM 多源导入 → Claude API 查询的信息架构
- 9 Module 研究框架模板
- 生成-审校分离的对抗式质量控制流程
- `pptx-research` Skill（已持久化，跨 Session 可用）
- `3-statements-ultra` Skill（自带 QC Suite）
