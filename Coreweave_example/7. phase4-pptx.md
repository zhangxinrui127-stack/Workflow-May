## 二、子阶段 4C：McKinsey 级视觉优化

### 2.1 用户反馈

用户上传了 13 张 McKinsey 报告截图与 AI 生成 PPT 截图对比，识别出 5 个系统性视觉问题：

> 1. Layout not centered — 内容整体偏上
> 2. Text overflow/overlap — 文本框过紧导致溢出
> 3. Table styling monotonous — 重色表头 + 机械交替行
> 4. Overall aesthetic — 可换字体（如 Book Antiqua）
> 5. Title-content hierarchy — 层次不够分明

### 2.2 Plan Mode 设计

Claude 进入 Plan Mode，分析 13 张截图，提炼 McKinsey 设计语言核心特征：

| McKinsey 特征 | AI Deck 现状 | 修正方案 |
|---------------|-------------|----------|
| 干净衬线标题 | Times New Roman 20pt | Book Antiqua 18pt |
| 无重色表头 | 深蓝填充+白色文字 | 白底+深蓝文字+底边线 |
| 仅横向细边框 | 四面边框+交替底色 | 仅底边线 0.5pt |
| 大量留白 | 内容堆积顶部 | 垂直居中重排 |
| 克制的色彩使用 | 多色图表 | 色彩精简 |

### 2.3 6 项修改实施

**Change 1 — 字体系统：** Times New Roman → Book Antiqua（标题），Arial → Book Antiqua（正文）。标题 20pt→18pt，block title 10pt→11pt。

**Change 2 — 布局重心下移：**

| 元素 | 修改前 y | 修改后 y |
|------|---------|---------|
| Top accent | 0.00 | 0.00 |
| Breadcrumb | 0.15 | 0.18 |
| **Title** | **0.42** | **0.50** |
| Content start | ~1.25 | **~1.40** |
| Source/Footer | 5.20 | **5.15** |

**Change 3 — 表格 McKinsey 化：**

```javascript
// 修改前 hdrCell：深蓝填充 + 白色文字
{ bold: true, color: "FFFFFF", fill: { color: "1B2A4A" } }

// 修改后 hdrCell：白底 + 深蓝文字 + 底边线
{ bold: true, color: "1B2A4A", fill: { color: "FFFFFF" },
  border: [none, none, { pt: 1.5, color: "1B2A4A" }, none] }
```

**Change 4 — 文本溢出防护：** 标题 h=0.55"（容纳 2 行 18pt），内容不超过 y=4.90。

**Change 5 — 层次体系：** Title 18pt bold → Outliner 10pt normal → Block title 11pt bold primary → L0 bullet 11pt bold → L1 sub-bullet 10pt normal → Source 8pt italic。

**Change 6 — Cover/TOC/Section Dividers：** 字体跟随 F.title 自动更新。

## 六、方法论复盘

### 6.1 分治渲染架构的优势

将 44 页 Deck 拆分为 6 个文件独立渲染，通过 `_render_full.js` 的 `eval()` 机制统一拼接：
- **并行开发**：不同章节可以在不同 Session 中迭代，互不干扰
- **增量调试**：单章报错不影响其他章节
- **设计系统统一**：master 设计系统定义在 `_render_full.js`，通过 eval() 上下文共享
- **独立预览**：每个 chapter 文件可独立运行生成单章 PPTX

### 6.2 数据一致性的脆弱性

本次审计发现的 15 个 Must-Fix 错误中，**12 个是级联错误**——P28 场景参数错误级联到 P39-P43 全部 5 个 thesis 页面。根因是 Model（Phase 5）在 Deck 初版渲染（Phase 4B）之后又做了参数调整，但 Deck 未同步。

**教训：** 当 Model 和 Deck 并行开发时，需要在 Model 终版确定后做一次全量同步。本次通过子阶段 4D 的交叉校验弥补了这一 gap。

### 6.3 根因追溯 vs 直接替换

用户要求"先查清为什么写错，再修改"。这一指令的价值在于：
- 发现 Memo 中 FY2025 Net Loss -$862M 实为 FY2024 数据（非笔误，而是版本混淆）
- 发现 D&A $2.1B 仅含 depreciation 而遗漏了 amortization（认知盲区，非计算错误）
- 确认 P28 场景参数使用了旧草稿版本（流程问题，非数据问题）

如果直接替换而不追溯，同一类错误可能在其他位置仍然存在。
